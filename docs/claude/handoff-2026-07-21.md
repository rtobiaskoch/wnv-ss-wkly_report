### The original bug

`calc_pir()` filtered `total > 1`, which dropped real single-mosquito pools — including
positives. CSU23480 (NW / 2025 / wk26 / Pipiens) was the audit's first instance.

- The filter had a legitimate half: `pIR()` **does** fail on `total == 0` rows with a
  non-NA `test_code` (`missing value where TRUE/FALSE needed`). That is the error the
  filter was originally written to prevent. `total == 1` was collateral — `> 1` where
  `> 0` would have done.
- Fixed to `filter(!is.na(test_code), total > 0)`, plus an explicit `nrow == 0` guard so
  the old `integerOneIndex` error can never resurface unexplained.
- The lines 73–93 single-pool fallback was deleted; it only existed to patch groups the
  filter had emptied, and would now duplicate rows.
- Impact: **12** false-zero cells corrected back to 2015; 327 further PIR cells shift by
  a median of 0.0001 (size-1 negatives now correctly in the denominator). Zero
  regressions — nothing goes `>0 → 0`.

## Notes

**Sharp edges discovered this session — these cost real time.**

1. **`config/run_config.sh` is hardcoded.** It ignores all arguments and always runs
   `--year 2025 --week 37 --download T`. It is the documented entry point in CLAUDE.md.
   Call `config/config_weekly.R` directly instead.

2. **The active config had `update = TRUE, push = TRUE`.** A render would have written to
   the master Google Sheet and pushed to git. It was regenerated as
   `2026 w29, download F, update F, push F`. **Re-run config before any live run.**

3. **`load_packages.R` used to reinstall wnvSurv from GitHub on every run**, silently
   reverting local fixes mid-session. This produced hours of "the fix keeps disappearing".
   Now fixed, but it means the **local install is authoritative** — a collaborator without
   the `../wnv-ss_functions` checkout gets whatever is on GitHub main.

4. **`utils/fun_calc_vi_stats.R` defined a function named `calc_vi`**, not
   `calc_vi_stats`. The real `calc_vi` only won by locale collation luck in
   `list.files()`. Archived. Watch for this pattern elsewhere in `utils/`.

5. **`trap_hx_combiner`'s test suite cannot run standalone** — no `helper-setup.R`.
   Use `devtools::load_all(".")` first or it reports spurious "could not find function".

6. **Positional diffs on cleaner output are meaningless** — implementations arrange
   differently. Stamp a `.rowid` on the input and realign, or compare row multisets.
   Three separate wrong conclusions were drawn before this was noticed.

7. **Unit tests were not sufficient.** Two real bugs passed the full suite and were only
   caught by diffing against real data:
   - a `trap_status` branch-order bug (label checked before counts, so a trap-night with
     a stray `No Mosquitoes` row alongside 122 *Culex tarsalis* was called empty);
   - `spp0` leaking into `zone_stats.csv` as an 18th column.
   Always diff the real outputs, including the schema.

8. **`clean_summary()` is `@keywords internal` in wnvSurv** — not exported. Nothing
   active calls it directly, but a `utils/` function that did would break.
